# 管理者とAIとサブエージェント——WordPressプラグインをチームで作る物語

> **著者**: yasuo3o3（管理者）＋ Claude（AI）
> **日付**: 2026-02-24
> **対象読者**: WordPress開発者、AI活用に興味のあるエンジニア、ノーコード・ローコードに疲れた人

---

## はじめに——「一人で全部やらなくていい」という発見

プラグイン開発は孤独な作業だと思っていた。

仕様を考え、コードを書き、テストし、WordPressの審査規約に合わせて直し——それを繰り返す。
しかし2026年、その常識は少し変わりつつある。

この記事は、**管理者（人間）とAI（Claude Code）とサブエージェント**という3つの存在が、どのように役割分担しながらWordPressプラグインを開発していくか、その実際の会話と仕組みを記録したものだ。

コードの話だけではない。「AIに何をさせるか」「自分は何を決めるか」という、**新しい開発スタイルの哲学的な話**でもある。

## 第一章：プロジェクトの構造——AIへの「憲法」を作る

### CLAUDE.md という存在

このプロジェクトには `CLAUDE.md` というファイルがある。
Claude Code はチャットを始めるたびにこのファイルを自動で読み込む。
いわば**「AIへの憲法」**だ。

```
CLAUDE.md の主な内容：
- どのドキュメントを、どの順番で読むか
- 作業前・中・後に何をドキュメント化するか
- 毎セッション終了時に HANDOFF.md を更新すること
```

これは単なるメモではない。**AIが毎回ゼロから学習し直さなくてもいいように**、プロジェクトの文脈を継承する仕組みだ。

```markdown
## 引き継ぎ確認
- `HANDOFF.md`（ルート）が存在する場合は最初に読み、現在の状況を把握せよ
- 前回のエージェントが残した「次にやるべきこと」を確認し、継続作業がある場合はそこから再開せよ
```

人間のチームでいう「朝会（スタンドアップ）」をAIが自動でこなすイメージだ。

---

### ドキュメントの階層構造

プロジェクトの規約は3層構造になっている：

```
WORDPRESS.md   ← 最上位規範（WordPress公式審査の要件）
    ↓
AGENTS.md      ← AIの行動規範（ワークフロー・出力ルール）
    ↓
CONTRIB.md     ← 環境固有のルール（担当者・環境設定）
```

**なぜこの構造が重要か？**

AIは「言われたことをやる」のが得意だが、「何を優先すべきか」の判断は苦手なことがある。
この階層を明示することで、「WordPressの審査規約 > AIの便利な提案」という優先順位をAIに徹底させている。

## 第二章：サブエージェントとは何か——AIの中のチーム

### 「このウィンドウからサブエージェントを立ち上げられますか？」

この日の会話はシンプルな質問から始まった。

> **管理者**: 「このウィンドウからサブエージェントを立ち上げ、何かの調査を取りまとめてもらうことができるのでしょうか？」

AIの回答は「はい、できます」。
しかし重要なのは「どうやって？」「何のために？」という部分だ。

### サブエージェントの種類と役割

Claude Code が起動できるサブエージェントには、それぞれ専門の役割がある：

```
┌─────────────────────────────────────────────────────┐
│                  Claude Code（メイン）                 │
│                   ↓  Task ツール  ↓                   │
├──────────┬──────────┬──────────┬──────────┬─────────┤
│ Explore  │  Plan    │  Bash    │ general  │  ...    │
│          │          │          │ purpose  │         │
│コードベース│実装計画の│コマンド  │複雑な    │         │
│の調査・  │設計・    │実行・    │マルチ    │         │
│探索      │アーキ    │Git操作   │ステップ  │         │
│          │テクチャ  │          │タスク    │         │
└──────────┴──────────┴──────────┴──────────┴─────────┘
```

### 並列調査の威力

サブエージェントの最大のメリットは**並列実行**だ。

通常のAIとの会話では、「Aを調べてから、次にBを調べる」という順番になる。
しかしサブエージェントを使うと：

```
メインAI: 「AとBを同時に調べてきて」
    ├── Exploreエージェント1: Aを調査中...
    └── Exploreエージェント2: Bを調査中...
         ↓ 両方完了
メインAI: 結果を統合して管理者に報告
```

これは**コンテキストウィンドウの節約**にもなる。
大量のファイルを読み込んでもメインのAIの「記憶」を圧迫せず、サブエージェントが処理した結果のサマリーだけを受け取れる。

## 第三章：WordPress プラグイン開発の実際——AIと規約の緊張関係

### このプロジェクトで作っているもの

`andw-parts-sketch` という名前が示すように、これは**WordPress用のパーツ（部品）管理プラグイン**だ。
接頭辞 `andw` はプロジェクト固有の命名規約で、すべての関数・クラス・定数に付ける。

### AI開発での「危険地帯」

WORDPRESS.md には明確な禁止事項が列挙されている：

```markdown
## 禁止事項
1. コア直読み禁止：wp-config.php / wp-load.php の直接 require 禁止
2. <script>/<style> 直書き禁止：JS/CSS は wp_enqueue_* のみ
3. php://input 丸読み禁止：$_POST 全量処理・全量ログ禁止
```

これらはすべて「AIが便利だと思ってやりがちなこと」でもある。

例えば、セキュリティを無視すれば：

```php
// AIが提案しがちな「簡単な」実装（NG例）
$data = $_POST; // 全量受け取り → 禁止
echo "<script>console.log('debug')</script>"; // 直書き → 禁止
```

しかし規約に従った正しい実装：

```php
// 正しい実装（セキュリティ・i18n・nonce すべて準拠）
if ( ! check_ajax_referer( 'andw_nonce', 'nonce', false ) ) {
    wp_send_json_error( [ 'message' => __( 'Invalid nonce.', 'andw-parts-sketch' ) ] );
    return;
}
$title = sanitize_text_field( wp_unslash( $_POST['title'] ?? '' ) );
```

AIはこの「正しい実装」を知っている。
しかし**規約ファイルで明示的に禁止事項を書いておくことで、AIが手を抜かない**ようにする——それがこのプロジェクトの仕組みだ。

## 第四章：作業停止ゲート——「待った」をかける仕組み

AGENTS.md には **Stop Gates（作業停止ゲート）** という概念がある：

```markdown
## 8. 作業停止ゲート (Stop Gates)
以下に該当する場合は直ちに作業を停止し、修正を提案すること：
- Text Domain ≠ スラッグ
- Contributors に申請アカウント名が無い
- <script>/<style> 直書き、コア直読み、$_POST 全量処理の指示
- WORDPRESS.md の禁止事項に触れる指示
```

これは「管理者の指示であっても止まれ」というルールだ。

**人間が間違えることもある**。「とりあえず動けばいい」と言いたくなる瞬間はある。
しかしWordPress.orgの審査で落ちれば、その「とりあえず」のツケは大きい。

AIに「止まれ」というゲートを設けることで、管理者の一時的な判断ミスをシステムが防いでくれる。
これは**AIによる品質保証の自動化**とも言える。

## 第五章：ドキュメント駆動開発——物語を記録する

### なぜドキュメントが先なのか

このプロジェクトでは、**実装前に計画書を書く**ことが義務付けられている：

```
実装前: docs/PHASE-PLAN.md に実装計画を作成（オーナー承認後に着手）
実装中: CHANGELOG.md をバージョンごとに更新
実装後: TESTING.md にテスト計画を作成
毎セッション終了時: HANDOFF.md を更新
```

これは「ウォーターフォール的」に見えるかもしれないが、**AIとの協働では特に重要**だ。

AIはセッションをまたいで記憶を持てない（基本的には）。
`HANDOFF.md` という「引き継ぎ書」を毎回更新することで、次のセッションのAIが「前回どこまでやったか」をゼロから把握できる。

### 会話ログの記録

`docs/conversation-log/YYYY-MM-DD.md` には、ユーザーの要求と最終回答だけを記録する。
途中経過の思考やコードは含めない。これはシンプルに：

- **なぜ**この機能を作ったのか
- **どんな質問**が意思決定の起点だったか

という「物語の骨格」を未来の自分（や他の開発者）のために残す行為だ。

## 第六章：この開発スタイルの哲学

### 「管理者とAI」の正しい関係

よくある誤解：「AIに全部やってもらえばいい」

実際の役割分担：

| 管理者（人間）がやること | AI がやること |
|------------------------|--------------|
| 何を作るかを決める | どうやって作るかを提案する |
| 規約の上位方針を設定する | 規約に沿った実装を行う |
| 最終的な承認をする | 承認前に問題点を洗い出す |
| ビジネス的な判断をする | 技術的な検証をする |
| ユーザーテストを行う | 静的チェック・構文チェックを行う |

AIは「高速に動く実装担当」であり、管理者は「方向を決めるディレクター」だ。

### サブエージェントが変えること

サブエージェントが加わることで、この構造はさらに一段進む：

```
管理者（ディレクター）
    ↓
メインAI（プロデューサー）
    ↓
サブエージェント群（専門スタッフ）
    ↓
コード・ドキュメント・テスト
```

プロデューサー（メインAI）は全体を俯瞰し、適切なスタッフ（サブエージェント）に仕事を割り振る。
ディレクター（管理者）は最終判断だけに集中できる。

---

## おわりに——物語はまだ続く

このプロジェクトは現在進行形だ。

`andw-parts-sketch` というWordPressプラグインは、管理者とAIとサブエージェントが少しずつコードを積み上げながら育っている。

重要なのは、**これが「AIに作ってもらった」プラグインではない**という点だ。
管理者が規約を設計し、禁止事項を定め、承認の権限を持ち続けている。
AIはその枠の中で最大限に動く。

「AIを使いこなす」とは、AIに自由を与えることではなく、**AIが最大限に力を発揮できる構造を人間が設計すること**なのかもしれない。
